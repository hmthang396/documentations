Phase 1 — Tại sao At-Least-Once?
Idempotency — nền tảng thay thế EOS
Với ALO, duplicate là chắc chắn xảy ra. Câu hỏi không phải "làm sao tránh duplicate" mà là "làm sao xử lý duplicate an toàn". Idempotency là câu trả lời.
Ba loại idempotency trong finance
A
Natural idempotency — SET thay vì INCREMENT
UPDATE account SET balance = 1000 WHERE id = X — chạy 100 lần kết quả như nhau
B
Event-ID dedup — INSERT IF NOT EXISTS
Dùng idempotency_key unique để check "đã xử lý chưa" trước khi write. Pattern phổ biến nhất.
C
Idempotency key — API level
Client gửi idempotency-key header. Server cache result và trả về cùng response nếu gọi lại.
Tại sao không chỉ dùng EOS để tránh duplicate?

Phase 2 — Architecture
Idempotency key schema — thiết kế trước khi code
Idempotency key phải được thiết kế cẩn thận. Nó là hợp đồng giữa producer và consumer để đảm bảo "cùng một operation" luôn được identify đúng, dù retry bao nhiêu lần.
public record PaymentEvent(
    // === Idempotency — BẮT BUỘC ===
    String eventId,          // UUID v4, unique per message (khác nhau mỗi retry)
    String idempotencyKey,   // Stable per operation — dùng để dedup
                             // Format: "{clientRef}:{operation}:{accountId}"
                             // Ví dụ: "CLIENT-789:DEBIT:ACC-456"

    // === Business ===
    String paymentId,        // Business ID — có thể có nhiều events
    String fromAccount,
    String toAccount,
    BigDecimal amount,
    String currency,

    // === Observability ===
    String correlationId,    // Trace qua nhiều services
    String causationId,      // eventId của event trigger event này
    Instant occurredAt,      // Business time (không phải Kafka timestamp)
    int version              // Schema version cho backward compat
) {}

// Build idempotency key — stable per business operation
public static String buildKey(String clientRef, String op, String accountId) {
    return clientRef + ":" + op + ":" + accountId;
    // Dù retry 100 lần, key này vẫn là một
    // → debit account ACC-456 với ref CLIENT-789 chỉ xảy ra 1 lần
}
eventId vs idempotencyKey: eventId = unique per message, đổi mỗi retry. idempotencyKey = stable per operation, giống nhau qua retry. Consumer dedup dùng idempotencyKey. Audit log dùng eventId để phân biệt từng delivery.

Phase 3 — Producer
Outbox Pattern — đảm bảo không mất event
Vấn đề: Service ghi DB thành công rồi crash trước khi publish Kafka → event bị mất. Outbox Pattern: ghi event vào DB cùng transaction với business data — atomic, không bao giờ mất.
1
API nhận request → begin DB transaction
Một transaction duy nhất bao gồm cả business write lẫn outbox write
2
Ghi business data (payments, accounts)
Business state thay đổi trong DB
3
INSERT outbox entry — cùng transaction
outbox(id, topic, key, payload, status=PENDING)
4
Commit DB transaction — atomic
Cả business data + outbox — cùng thành công hoặc cùng rollback
5
Outbox Worker poll → Kafka publish → mark PUBLISHED
Background worker, retry nếu Kafka unavailable
@Transactional  // DB transaction
public PaymentResponse processPayment(PaymentRequest req) {
    // Business writes
    Payment payment = paymentRepo.save(new Payment(req));
    accountRepo.debit(req.fromAccount(), req.amount());

    // Outbox write — cùng DB transaction
    outboxRepo.save(OutboxEntry.builder()
        .topic("payment-requests")
        .partitionKey(req.fromAccount())  // ordering per account
        .payload(toJson(PaymentEvent.of(payment)))
        .idempotencyKey(payment.getIdempotencyKey())
        .status("PENDING")
        .build());

    return PaymentResponse.of(payment);
    // DB commit: payment + account + outbox — atomic
}
@Scheduled(fixedDelay = 100)
public void publishOutbox() {
    List<OutboxEntry> pending = outboxRepo.findTop100Pending();
    for (OutboxEntry e : pending) {
        try {
            producer.send(buildRecord(e)).get(10, TimeUnit.SECONDS);
            outboxRepo.markPublished(e.getId());
        } catch (Exception ex) {
            outboxRepo.scheduleRetry(e.getId(), computeBackoff(e));
        }
    }
}
Kết quả: Dù service crash ở bất kỳ điểm nào, outbox table vẫn có record PENDING → Worker sẽ publish khi restart. Duplicate có thể xảy ra → consumer phải dedup.
Phase 3 — Producer
Outbox retry — exponential backoff + alerting
Outbox Worker phân biệt transient vs permanent failure. Không retry vô hạn — sau MAX_RETRIES chuyển sang FAILED và alert ops. Mọi failed message đều có trace để investigate.
private static final int MAX_RETRIES = 10;
// Retry schedule (ms): 200, 400, 800, 1600, 3200...
// cap tại 60_000ms sau ~8 lần retry

private void publish(OutboxEntry entry) {
    try {
        producer.send(buildRecord(entry)).get(10, TimeUnit.SECONDS);
        outboxRepo.markPublished(entry.getId());

    } catch (SerializationException e) {
        // Permanent — không retry
        outboxRepo.markFailed(entry.getId(), e.getMessage());
        alerting.critical("Outbox permanent failure: " + entry.getId());

    } catch (Exception e) {
        // Transient — exponential backoff
        int retries = entry.getRetryCount() + 1;

        if (retries >= MAX_RETRIES) {
            outboxRepo.markFailed(entry.getId(), "Max retries");
            alerting.critical("Outbox max retries: " + entry.getId());
        } else {
            long backoffMs = Math.min(
                (long) Math.pow(2, retries) * 100,
                60_000
            );
            outboxRepo.scheduleRetry(
                entry.getId(), retries,
                Instant.now().plusMillis(backoffMs)
            );
        }
    }
}
Outbox table schema cần: retry_count INT, next_retry_at TIMESTAMPTZ, last_error TEXT, status VARCHAR. Index trên (status, next_retry_at) để poll hiệu quả không full scan.
Phase 4 — Consumer + Dedup
Consumer config — ALO với manual commit
enable.auto.commit=false và commit sau xử lý là điểm cốt lõi của ALO. Commit trước = at-most-once (có thể mất). Commit sau = at-least-once (có thể duplicate, cần dedup).
Properties props = new Properties();

// === ALO bắt buộc ===
props.put("enable.auto.commit", false);
// Auto-commit commit SAU poll() — nếu crash khi xử lý
// offset đã committed → message bị mất → at-most-once!

props.put("isolation.level", "read_uncommitted");
// OK vì Outbox Worker không dùng Kafka transactions

props.put("auto.offset.reset", "earliest");
// Finance: không bao giờ bỏ qua messages

// === Tuning cho finance ===
props.put("max.poll.records", 100);
// Nhỏ hơn thông thường — mỗi record cần DB write + dedup check
// 100 records × ~5ms/record = 500ms processing time

props.put("max.poll.interval.ms", 300_000); // 5 phút
props.put("session.timeout.ms", 45_000);
props.put("heartbeat.interval.ms", 3_000);
Anti-pattern: Đừng dùng commitAsync() làm commit chính — không throw exception khi fail, bạn không biết offset đã committed chưa. Dùng commitSync() trong vòng lặp chính. commitAsync() chỉ trong shutdown hook.
Phase 4 — Consumer + Dedup
Dedup Layer — Database idempotency check
Layer dedup chính cho finance: check idempotencyKey đã xử lý chưa trước khi chạy bất kỳ business logic nào. Ghi kết quả vào processed_events cùng DB transaction với business write.
-- processed_events table
CREATE TABLE processed_events (
    idempotency_key  VARCHAR(255) PRIMARY KEY,
    event_id         UUID NOT NULL,
    result_payload   TEXT,           -- trả về nếu retry
    processed_at     TIMESTAMPTZ NOT NULL,
    consumer_group   VARCHAR(100) NOT NULL,
    kafka_topic      VARCHAR(255) NOT NULL,
    kafka_partition  INT NOT NULL,
    kafka_offset     BIGINT NOT NULL  -- để trace duplicate
);
@Transactional
public ProcessResult processWithDedup(
        PaymentEvent event, ConsumerRecord<?,?> record) {

    // Step 1: Check dedup — idempotencyKey là chìa khóa
    Optional<ProcessedEvent> existing =
        processedEventRepo.findById(event.idempotencyKey());

    if (existing.isPresent()) {
        log.info("[DEDUP] Skip duplicate: key={}, originalOffset={}",
                 event.idempotencyKey(), existing.get().getKafkaOffset());
        metrics.dupSkipped.increment();
        return ProcessResult.fromExisting(existing.get());
    }

    // Step 2: Business logic (chưa xử lý trước đây)
    PaymentResult result = executePayment(event);

    // Step 3: Ghi processed_events + business data — cùng transaction
    processedEventRepo.save(ProcessedEvent.builder()
        .idempotencyKey(event.idempotencyKey())
        .eventId(event.eventId())
        .resultPayload(toJson(result))
        .processedAt(Instant.now())
        .consumerGroup("payment-processor")
        .kafkaTopic(record.topic())
        .kafkaPartition(record.partition())
        .kafkaOffset(record.offset())
        .build());

    return ProcessResult.of(result);
    // Commit DB: business data + processed_events — atomic
}
UPSERT tốt hơn SELECT + INSERT: Dùng INSERT INTO processed_events ... ON CONFLICT (idempotency_key) DO NOTHING. Atomic hơn, tránh race condition khi concurrent consumers xử lý cùng message trong cùng lúc.
Phase 4 — Consumer + Dedup
Dedup Layer — Redis cache (notification service)
Notification không cần độ bền như payment. Redis TTL-based dedup đủ dùng và nhanh hơn nhiều. Duplicate trong window TTL bị blocked — sau đó có thể gửi lại (acceptable với notifications).
public class NotificationDeduplicator {
    private final RedisTemplate<String, String> redis;
    private static final Duration TTL = Duration.ofHours(24);

    public boolean tryMarkProcessed(String idempotencyKey) {
        String key = "notif:dedup:" + idempotencyKey;

        // SET NX EX — atomic: chỉ set nếu chưa tồn tại
        Boolean absent = redis.opsForValue().setIfAbsent(key, "1", TTL);

        if (Boolean.TRUE.equals(absent)) {
            return true;  // chưa có → xử lý
        }
        log.info("[DEDUP] Notif dup skipped: {}", idempotencyKey);
        metrics.notifDupSkipped.increment();
        return false; // đã có → skip
    }
}

// Usage trong consumer loop
for (ConsumerRecord<?,?> record : records) {
    PaymentResultEvent event = deserialize(record);
    if (!deduplicator.tryMarkProcessed(event.idempotencyKey()))
        continue;
    notificationService.send(event);
}

Phase 4 — Consumer + Dedup
Idempotent DB write — ledger entry
Business write trong payment processor phải idempotent ở DB level. INSERT ON CONFLICT DO NOTHING là cơ chế cuối cùng để đảm bảo dù dedup check bị race condition, ledger vẫn không bị double-entry.
-- ledger_entries — enforce idempotency tại DB level
CREATE TABLE ledger_entries (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    idempotency_key  VARCHAR(255) UNIQUE NOT NULL, -- DB constraint
    account_id       VARCHAR(50)  NOT NULL,
    entry_type       VARCHAR(10)  NOT NULL,  -- DEBIT / CREDIT
    amount           DECIMAL(20,4) NOT NULL,
    currency         CHAR(3)      NOT NULL,
    payment_id       VARCHAR(50)  NOT NULL,
    posted_at        TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
@Transactional
public LedgerResult postEntries(PaymentEvent event) {

    // ON CONFLICT DO NOTHING — idempotent at DB level
    // Nếu idempotency_key đã tồn tại (duplicate): không insert, không exception
    int debitRows = jdbcTemplate.update("""
        INSERT INTO ledger_entries
            (idempotency_key, account_id, entry_type, amount, currency, payment_id)
        VALUES (?, ?, 'DEBIT', ?, ?, ?)
        ON CONFLICT (idempotency_key) DO NOTHING
        """,
        event.idempotencyKey() + ":DEBIT",
        event.fromAccount(), event.amount(), event.currency(), event.paymentId()
    );

    int creditRows = jdbcTemplate.update("""
        INSERT INTO ledger_entries
            (idempotency_key, account_id, entry_type, amount, currency, payment_id)
        VALUES (?, ?, 'CREDIT', ?, ?, ?)
        ON CONFLICT (idempotency_key) DO NOTHING
        """,
        event.idempotencyKey() + ":CREDIT",
        event.toAccount(), event.amount(), event.currency(), event.paymentId()
    );

    if (debitRows == 0 && creditRows == 0) {
        log.info("[DEDUP] Ledger already posted: {}", event.idempotencyKey());
        return LedgerResult.existing(fetchExisting(event.idempotencyKey()));
    }
    return LedgerResult.posted(debitRows, creditRows);
}
Phase 4 — Consumer + Dedup
Offset commit — sau khi xử lý thành công
Vòng lặp chính của consumer: poll → dedup → business logic → commit offset. Commit SAU là ALO. Nếu crash trước commit → reprocess → dedup sẽ handle duplicate.
while (running) {
    ConsumerRecords<String, String> records =
        consumer.poll(Duration.ofMillis(500));
    if (records.isEmpty()) continue;

    boolean batchSuccess = true;

    for (ConsumerRecord<String, String> record : records) {
        try {
            PaymentEvent event = deserialize(record.value());
            processor.processWithDedup(event, record); // dedup + business + DB

        } catch (DeserializationException | BusinessException e) {
            // Permanent failure → DLQ, tiếp tục batch
            dlqPublisher.send(record, e, FailureType.PERMANENT);

        } catch (TransientException e) {
            // DB timeout, network blip → KHÔNG commit offset
            // Toàn bộ batch sẽ được reprocess → dedup handle
            batchSuccess = false;
            retryHandler.backoff(e);
            break;
        }
    }

    if (batchSuccess) {
        // Commit ONLY sau khi batch xử lý xong
        // (thành công hoặc permanent failures đã DLQ)
        consumer.commitSync();
    }
    // !batchSuccess: không commit → reprocess từ đầu batch
}
Granular commit: Với batch lớn, commit per-record thay vì per-batch. Dùng consumer.commitSync(Map<TopicPartition, OffsetAndMetadata>) sau mỗi record thành công. Giảm reprocess window khi có lỗi giữa batch.
Phase 5 — Finance specifics
Double-entry ledger — balance verification
Finance yêu cầu double-entry: mỗi transaction phải có DEBIT = CREDIT. Consumer không chỉ ghi ledger mà còn verify balance sau mỗi batch và chạy reconciliation job định kỳ.
@Transactional(readOnly = true)
public BalanceVerification verifyBalance(String accountId) {
    BigDecimal accountBalance = accountRepo.getBalance(accountId);

    // Compute từ ledger
    BigDecimal ledgerBalance = jdbcTemplate.queryForObject("""
        SELECT SUM(
            CASE WHEN entry_type='CREDIT' THEN amount ELSE -amount END
        )
        FROM ledger_entries WHERE account_id = ?
        """, BigDecimal.class, accountId);

    boolean balanced = accountBalance.compareTo(ledgerBalance) == 0;

    if (!balanced) {
        // CRITICAL alert ngay lập tức
        alerting.critical("LEDGER IMBALANCE", Map.of(
            "accountId", accountId,
            "accountBalance", accountBalance,
            "ledgerBalance", ledgerBalance,
            "diff", accountBalance.subtract(ledgerBalance)
        ));
    }
    return new BalanceVerification(accountId, balanced, ledgerBalance);
}

// Reconciliation job — chạy mỗi giờ
@Scheduled(cron = "0 0 * * * *")
public void reconcileAll() {
    // Kiểm tra tổng toàn hệ thống: SUM(DEBIT) == SUM(CREDIT)
    // Bất kỳ imbalance → INCIDENT
    BigDecimal totalDebits  = ledgerRepo.sumByType("DEBIT");
    BigDecimal totalCredits = ledgerRepo.sumByType("CREDIT");
    if (totalDebits.compareTo(totalCredits) != 0)
        alerting.incident("SYSTEM LEDGER IMBALANCE", totalDebits, totalCredits);
}
Phase 5 — Finance specifics
Saga pattern — compensating transactions
Multi-step payment (debit → transfer → credit) cần Saga. Nếu step sau fail, các steps trước phải được compensate bằng reverse entries — không xóa entry cũ, chỉ ghi entry ngược chiều.
// Step 1: Nhận payment-requests → debit
public void onPaymentRequested(PaymentEvent event) {
    if (!dedup.check(event.idempotencyKey() + ":DEBIT")) return;
    try {
        ledger.debit(event.fromAccount(), event.amount());
        publisher.publish("payment-debited", PaymentDebitedEvent.of(event));
    } catch (InsufficientFundsException e) {
        publisher.publish("payment-failed", FailedEvent.of(event, "INSUFFICIENT"));
    }
}

// Step 2: Nhận payment-debited → credit
public void onPaymentDebited(PaymentDebitedEvent event) {
    if (!dedup.check(event.idempotencyKey() + ":CREDIT")) return;
    try {
        ledger.credit(event.toAccount(), event.amount());
        publisher.publish("payment-completed", CompletedEvent.of(event));
    } catch (Exception e) {
        // Credit fail → cần compensate debit
        publisher.publish("payment-credit-failed", CreditFailedEvent.of(event, e));
    }
}

// Compensation: Nhận payment-credit-failed → reverse debit
public void onCreditFailed(CreditFailedEvent event) {
    String key = event.idempotencyKey() + ":DEBIT:REVERSAL";
    if (!dedup.check(key)) return;

    // Reverse entry — KHÔNG xóa entry cũ (audit trail)
    ledger.credit(event.fromAccount(), event.amount());
    auditLog.write("PAYMENT_REVERSED", event);
    publisher.publish("payment-reversed", ReversedEvent.of(event));
}
Idempotency trong Saga: Mỗi step có key riêng (:DEBIT, :CREDIT, :DEBIT:REVERSAL). Dù step được retry bao nhiêu lần, chỉ một lần được thực thi. Compensation cũng phải idempotent.

Dead Letter Queue — finance grade
Finance DLQ cần đủ thông tin để ops team quyết định: retry, manual fix, hay escalate. Không "just drop it" — mọi failed message phải có audit trail và human review trước khi replay.
public void send(ConsumerRecord<?,?> original, Exception reason, FailureType type) {
    Headers headers = new RecordHeaders()
        .add("dlq-failure-type",      type.name().getBytes())
        .add("dlq-reason",             truncate(reason.getMessage()).getBytes())
        .add("dlq-exception-class",    reason.getClass().getName().getBytes())
        .add("dlq-original-topic",     original.topic().getBytes())
        .add("dlq-original-partition",  intToBytes(original.partition()))
        .add("dlq-original-offset",     longToBytes(original.offset()))
        .add("dlq-consumer-group",      groupId.getBytes())
        .add("dlq-timestamp",           Instant.now().toString().getBytes());

    producer.send(new ProducerRecord<>(
        "payment-dead-letter",
        null, original.key(), original.value(), headers
    ));

    log.error("[DLQ] type={} topic={} offset={} key={}",
              type, original.topic(), original.offset(), original.key());
}

public enum FailureType {
    DESERIALIZATION_ERROR,  // payload corrupt
    VALIDATION_ERROR,       // business rule violation
    ACCOUNT_NOT_FOUND,      // reference data issue
    COMPLIANCE_REJECT,      // fraud / AML block
    PERMANENT_SYSTEM_ERROR  // downstream permanently unavailable
}

@Testcontainers
class PaymentProcessorAloTest {

    @Test
    void duplicate_message_does_not_double_debit() {
        PaymentEvent event = buildPaymentEvent("ACC-A", "ACC-B", 100.00);

        // Publish TWICE — simulating ALO duplicate delivery
        publishToKafka(event); // first delivery
        publishToKafka(event); // duplicate

        await().atMost(10, SECONDS).until(() ->
            processedEventRepo.existsById(event.idempotencyKey()));

        // SINGLE ledger entry, not double
        assertThat(ledgerRepo.countByKey(event.idempotencyKey() + ":DEBIT"))
            .isEqualTo(1);  // NOT 2

        // Balance correct
        assertThat(accountRepo.getBalance("ACC-A"))
            .isEqualByComparingTo("900.00");  // NOT 800.00
    }

    @Test
    void crash_before_commit_reprocesses_safely() {
        PaymentEvent event = buildPaymentEvent(...);
        publishToKafka(event);

        // Process but "crash" before commitSync()
        processor.processRecord(event); // business done, DB committed
        // DO NOT call commitSync() — simulate crash

        // Restart — reprocesses from last committed offset
        processor.restart();
        processor.processRecord(event); // duplicate delivery
        processor.commitSync();

        // Still single ledger entry — dedup saved us
        assertThat(ledgerRepo.countByPaymentId(event.paymentId()))
            .isEqualTo(2); // DEBIT + CREDIT, not 4
        verifyBalanced("ACC-A", "ACC-B");
    }

    @Test
    void outbox_retries_when_kafka_unavailable() {
        kafka.stop();
        paymentApi.initiatePayment(buildRequest());
        assertThat(outboxRepo.countByStatus("PENDING")).isEqualTo(1);

        kafka.start();
        await().atMost(30, SECONDS).until(() ->
            outboxRepo.countByStatus("PUBLISHED") == 1);
    }
}
