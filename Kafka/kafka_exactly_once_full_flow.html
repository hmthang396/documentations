Phase 1 — Setup
Broker configuration
Trước khi viết một dòng code nào, broker phải được cấu hình đúng. EOS phụ thuộc hoàn toàn vào __transaction_state topic — nếu topic này mất data, toàn bộ transaction state biến mất.
Critical: Nếu transaction.state.log.replication.factor=1 và broker chết, mọi transaction đang chờ đều bị mất state. Production bắt buộc = 3.
server.properties
# Transaction state — đây là "source of truth" cho mọi transaction
transaction.state.log.replication.factor=3   # min cho production
transaction.state.log.min.isr=2

# Offset log
offsets.topic.replication.factor=3
offsets.topic.num.partitions=50

# Transaction timeout ceiling — producer's timeout phải <= giá trị này
transaction.max.timeout.ms=900000           # 15 phút

# Giữ state của transactional.id đã hoàn thành 7 ngày
transactional.id.expiration.ms=604800000

# ISR lag tối đa trước khi replica bị đuổi khỏi ISR
replica.lag.time.max.ms=30000
Verify: kafka-topics.sh --describe --topic __transaction_state — đảm bảo ReplicationFactor=3 và ISR có đủ 3 brokers trước khi deploy.
Phase 1 — Setup
Topic setup
Các topic data cũng cần min.insync.replicas=2. Nếu không, acks=all chỉ cần leader ghi là đủ — vô nghĩa với EOS.
# Input topic
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --create --topic payment-requests \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --config retention.ms=604800000   # 7 ngày

# Output topics
kafka-topics.sh --create --topic payment-processed \
  --partitions 12 --replication-factor 3 \
  --config min.insync.replicas=2

kafka-topics.sh --create --topic payment-failed \
  --partitions 12 --replication-factor 3 \
  --config min.insync.replicas=2

# Dead letter queue
kafka-topics.sh --create --topic payment-dead-letter \
  --partitions 12 --replication-factor 3
Lưu ý: min.insync.replicas phải được set ở cả topic level. Broker-level default không đủ nếu topic được tạo trước khi thay đổi default.

Idempotent producer config
enable.idempotence=true tự động kéo theo acks=all, retries=MAX_INT, và giới hạn max.in.flight=5. Đây là nền tảng — transaction build trên idempotence.

// === EOS bắt buộc ===

// Tự động set: acks=all, retries=MAX_INT, max.in.flight=5

Phase 2 — Producer
Transactional ID — unique per instance
Đây là lỗi phổ biến nhất. Hai instances cùng transactional.id sẽ fence lẫn nhau liên tục. Mỗi pod/instance phải có ID riêng biệt và ổn định trong suốt lifetime của nó.
static String generateTransactionalId(String servicePrefix) {
    // Ưu tiên dùng pod name (Kubernetes inject vào env)
    String podName = System.getenv("POD_NAME");

    if (podName != null && !podName.isEmpty()) {
        // payment-processor-pod-abc123-xyz
        return servicePrefix + "-" + podName;
    }

    // Fallback: hostname (VM/bare metal deployment)
    try {
        String host = InetAddress.getLocalHost().getHostName();
        return servicePrefix + "-" + host;
    } catch (UnknownHostException e) {
        // Last resort: UUID (mỗi restart sẽ tạo ID mới — OK nhưng
        // transaction state cũ không bao giờ được claim lại)
        return servicePrefix + "-" + UUID.randomUUID();
    }
}

// Kết quả ví dụ: "payment-processor-payment-pod-7d4f9b-xkp2q"
Không được: Dùng random UUID mới mỗi lần startup. Mỗi ID mới = coordinator gán Epoch mới = transaction state cũ bị bỏ. Quá nhiều ID bỏ xó sẽ làm __transaction_state phình to.
Tốt nhất: POD_NAME từ Kubernetes downward API — stable trong lifetime của pod, thay đổi khi pod được restart/reschedule.
Phase 2 — Producer
initTransactions() — gọi một lần khi startup
initTransactions() làm 3 việc: tìm Transaction Coordinator, đăng ký PID + Epoch, và abort mọi transaction dở dang từ lần chạy trước. Phải gọi trước mọi beginTransaction().
public class PaymentProcessor {

    private final KafkaProducer<String, String> producer;
    private final KafkaConsumer<String, String> consumer;

    public void start() {
        // initTransactions() phải là bước ĐẦU TIÊN, trước subscribe()
        producer.initTransactions();
        // Lúc này: PID và Epoch đã được gán
        // Mọi transaction dở dang từ trước đã bị abort bởi coordinator

        consumer.subscribe(List.of("payment-requests"));

        // Bắt đầu processing loop...
        while (running) {
            processOneBatch();
        }
    }

    // initTransactions() sẽ throw nếu:
    // - Coordinator không phản hồi → TimeoutException
    // - Broker version cũ → UnsupportedVersionException
    // - ACL không đủ → AuthorizationException
}
Timing: initTransactions() là blocking call. Nó chờ coordinator phản hồi. Set max.block.ms nếu cần timeout rõ ràng khi coordinator chậm.
Phase 3 — Consumer
isolation.level = read_committed
Đây là config consumer bị quên nhiều nhất. Nếu thiếu, consumer đọc cả messages của transactions chưa committed — kể cả messages sẽ bị ABORT. Toàn bộ EOS guarantee sụp đổ phía consumer.
Properties props = new Properties();

// === EOS bắt buộc ===
props.put("isolation.level", "read_committed");
// Chỉ trả về messages của committed transactions
// + non-transactional messages
// Consumer blocked tại LSO (Last Stable Offset)

props.put("enable.auto.commit", false);
// Auto-commit không biết về transaction boundaries
// Dùng sendOffsetsToTransaction() thay thế

// === Tuning ===
props.put("max.poll.records", 500);
props.put("max.poll.interval.ms", 300_000);
// PHẢI > thời gian xử lý 1 batch worst case
// Nếu vượt quá → consumer bị evict → rebalance → reprocess

props.put("session.timeout.ms", 30_000);
props.put("heartbeat.interval.ms", 3_000);
// heartbeat < session.timeout / 3
LSO: Consumer với read_committed không thể đọc vượt quá Last Stable Offset. Nếu có transaction sống lâu chưa commit, consumer bị "chặn" tại đó dù có messages mới hơn đã committed ở phía sau.
Phase 3 — Consumer
Manual offset — không dùng commitSync()
Trong EOS flow, offset commit là một phần của transaction — không phải action riêng biệt. Dùng sendOffsetsToTransaction() thay vì commitSync().
// ❌ SAI — offset commit tách rời khỏi transaction
consumer.commitSync();  // Nếu transaction sau đó abort, offset đã committed
                        // → Consumer không reprocess → Data mất

// ✅ ĐÚNG — offset commit nằm trong transaction
producer.sendOffsetsToTransaction(
    buildOffsetMap(records),
    consumer.groupMetadata()  // Quan trọng: bao gồm generation ID
);
// Nếu transaction abort → offset không commit → consumer reprocess
// Nếu transaction commit → offset commit cùng lúc → atomic!

// Build offset map: partition → last_offset + 1
private Map<TopicPartition, OffsetAndMetadata> buildOffsetMap(
        ConsumerRecords<?, ?> records) {
    Map<TopicPartition, OffsetAndMetadata> map = new HashMap<>();
    for (TopicPartition tp : records.partitions()) {
        var recs = records.records(tp);
        long last = recs.get(recs.size() - 1).offset();
        map.put(tp, new OffsetAndMetadata(last + 1));
    }
    return map;
}
groupMetadata() vs groupId string: Dùng consumer.groupMetadata() (không phải string groupId) để bao gồm generation ID. Nếu rebalance xảy ra trong lúc transaction đang chạy, coordinator phát hiện generation mismatch và reject offset commit — transaction bị abort, tránh double-processing.
Phase 4 — Transaction loop
Business logic ngoài transaction window
Nguyên tắc cốt lõi: transaction càng ngắn càng tốt. Chỉ đưa vào transaction những gì cần atomic — ghi outputs và commit offset. Business logic, validation, external calls đều làm trước.
private void processOneBatch() {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));
    if (records.isEmpty()) return;

    // ── NGOÀI TRANSACTION ──────────────────────────────────────────
    // Bước 1: Deserialize (có thể fail → đưa vào DLQ)
    List<PaymentRequest> parsed = deserialize(records);

    // Bước 2: Business logic — tốn thời gian, gọi external service
    // Đây có thể mất vài giây → OK vì chưa trong transaction
    List<PaymentResult> results = parsed.stream()
        .map(this::validateAndProcess)   // gọi account service, fraud check...
        .collect(toList());

    // ── TRONG TRANSACTION (ngắn, chỉ I/O Kafka) ────────────────────
    producer.beginTransaction();
    try {
        // Bước 3: Ghi results
        // Bước 4: Commit offset
        // → xem các panel tiếp theo
        producer.commitTransaction();
    } catch (KafkaException e) {
        producer.abortTransaction();
        throw new RetryableException(e);
    }
}
Tại sao tách biệt? Transaction có transaction.timeout.ms = 60s. Nếu business logic mất 30s, chỉ còn 30s để ghi + commit. Tách ra: business logic bao lâu cũng được, transaction chỉ mất 10-50ms.

beginTransaction()

// Từ lúc này, mọi producer.send() đều thuộc transaction này
// Coordinator biết: TxID=my-service-pod-abc → ONGOING
// Các partitions sẽ được thêm vào danh sách khi send() đầu tiên
// đến mỗi partition mới (AddPartitionsToTxn RPC)

Phase 4 — Transaction loop
send() — ghi output messages
Mỗi send() trong transaction window ghi message vào broker với trạng thái UNCOMMITTED. Consumer với read_committed không thấy những messages này cho đến khi transaction commit.
for (PaymentResult result : results) {
    String topic = result.success()
        ? "payment-processed"
        : "payment-failed";

    producer.send(
        new ProducerRecord<>(
            topic,
            result.requestId(),   // key = requestId → ordering per payment
            toJson(result)        // value = serialized result
        ),
        (metadata, exception) -> {
            // Callback chạy async — exception ở đây sẽ được
            // re-thrown khi commitTransaction() được gọi
            if (exception != null) {
                log.error("Send failed: {}", exception.getMessage());
            }
        }
    );
    // send() là non-blocking — return ngay lập tức
    // Actual network I/O xảy ra trong background thread
}

// Tại thời điểm này trong partition log:
// [BEGIN TxID=my-service-pod-abc]
// [Message M1, PID=12345, Seq=0, UNCOMMITTED]
// [Message M2, PID=12345, Seq=1, UNCOMMITTED]
Dedup tự động: Nếu network timeout và producer retry, broker kiểm tra PID+Seq. Nếu đã có Seq=0 → drop duplicate silently, trả ACK như bình thường. Không cần code extra.
Phase 4 — Transaction loop
sendOffsetsToTransaction() — atomic offset commit
Trái tim của EOS. Ghi offset vào __consumer_offsets như một phần của transaction — không phải action riêng. Nếu transaction abort, offset cũng không commit.
Map<TopicPartition, OffsetAndMetadata> offsets = buildOffsetMap(records);

producer.sendOffsetsToTransaction(
    offsets,
    consumer.groupMetadata()
    // groupMetadata() = {groupId, generationId, memberId}
    // generationId phát hiện rebalance trong quá trình này
);

// Điều gì xảy ra bên dưới:
// 1. Producer gửi TxnOffsetCommit request đến Group Coordinator
// 2. Group Coordinator ghi offset vào __consumer_offsets
//    với flag isTransactional=true
// 3. Offset chỉ có hiệu lực khi transaction commit
// 4. Nếu abort → offset bị rollback

// Nếu rebalance xảy ra TRONG lúc này:
// generationId trong groupMetadata() sẽ không match
// → Group Coordinator reject → Exception → abort transaction
// → Consumer mới sẽ reprocess từ last committed offset → OK
Guarantee: Offset commit và message write là một atomic unit. Không bao giờ có tình trạng "message đã ghi ra output topic nhưng offset chưa commit" hoặc ngược lại.
Phase 4 — Transaction loop
commitTransaction() — 2PC in action
Một lời gọi đơn giản nhưng trigger cả giao thức 2-Phase Commit. Sau khi return, mọi messages và offset commits đều durable và visible với consumers.
producer.commitTransaction();
// Đây là blocking call — chờ cho đến khi 2PC hoàn thành

// Bên dưới, giao thức 2PC diễn ra:
// Phase 1:
//   Producer → Coordinator: EndTxn(COMMIT)
//   Coordinator ghi PREPARE_COMMIT vào __transaction_state
//   → Irrevocable: từ đây không thể abort
//
// Phase 2:
//   Coordinator ghi COMMIT marker vào mỗi partition
//   Coordinator ghi COMMIT marker vào __consumer_offsets
//   Coordinator cập nhật __transaction_state: COMPLETE
//   Coordinator trả về OK cho Producer
//
// Sau commit:
//   Messages vượt qua LSO → consumer read_committed thấy được
//   Offset có hiệu lực → consumer group sẽ không reprocess
1
EndTxn(COMMIT) → Coordinator
Producer gửi intent commit
2
PREPARE_COMMIT ghi vào __transaction_state
Durability checkpoint — coordinator crash cũng không mất
3
COMMIT marker ghi vào mỗi partition
Messages chuyển từ UNCOMMITTED → COMMITTED
4
COMMIT marker ghi vào __consumer_offsets
Offset chính thức có hiệu lực
5
Consumer thấy messages (LSO advance)
read_committed consumer nhận batch
Phase 5 — Error handling
ProducerFencedException — fatal, shutdown ngay
Xảy ra khi producer khác đã claim cùng transactional.id với Epoch cao hơn. Instance hiện tại là "zombie" — không có quyền ghi bất cứ thứ gì. Phải shutdown ngay lập tức.
try {
    producer.beginTransaction();
    // ... send, sendOffsets ...
    producer.commitTransaction();

} catch (ProducerFencedException | InvalidProducerEpochException e) {
    // ZOMBIE FENCING:
    // Epoch của mình (3) < current epoch (4)
    // Instance khác đã restart với cùng transactional.id
    // và được gán Epoch mới = 4

    log.error("[FATAL] Producer fenced! transactional.id={}, "
            + "This is a zombie instance. Shutting down.",
              transactionalId, e);

    // KHÔNG được gọi abortTransaction() — không còn quyền
    // KHÔNG được retry — sẽ tiếp tục bị fence

    producer.close();              // đóng producer
    throw new FatalException(e);  // để caller shutdown service

} catch (AuthorizationException e) {
    // ACL issue — cũng fatal, không retry
    producer.close();
    throw new FatalException(e);
}
Zombie Fencing là feature, không phải bug. Nếu deployment cũ và mới chạy song song với cùng transactional.id, Kafka tự động đảm bảo chỉ có một instance "thắng". Instance cũ buộc phải dừng.
Phase 5 — Error handling
KafkaException — abort và retry
Lỗi có thể retry được (network timeout, broker không sẵn sàng tạm thời). Abort transaction hiện tại, reset consumer về last committed offset, và retry batch trong lần poll tiếp theo.
try {
    producer.beginTransaction();
    // ... send, sendOffsets ...
    producer.commitTransaction();
    retryHandler.recordSuccess();

} catch (ProducerFencedException e) {
    // Fatal — xem panel trước
    throw ...;

} catch (KafkaException e) {
    // Retryable — abort và thử lại
    log.warn("Transaction failed, aborting. Retry next poll.", e);

    try {
        producer.abortTransaction();
        // Coordinator nhận ABORT → ghi ABORT marker vào partitions
        // Messages đã send() bị ẩn khỏi consumers
    } catch (KafkaException abortEx) {
        // abort() cũng fail → state không rõ → shutdown an toàn
        log.error("abortTransaction() also failed", abortEx);
        producer.close();
        throw new FatalException(abortEx);
    }

    // Reset consumer về last committed offset
    resetToLastCommitted();

    // Exponential backoff
    retryHandler.handleRetryableFailure(e);
    // → sleep 2s, 4s, 8s... tối đa 5 lần rồi fatal
}
private void resetToLastCommitted() {
    for (TopicPartition tp : consumer.assignment()) {
        OffsetAndMetadata committed = consumer
            .committed(Set.of(tp)).get(tp);
        if (committed != null) {
            consumer.seek(tp, committed.offset());
        } else {
            consumer.seekToBeginning(Set.of(tp));
        }
    }
}
Phase 5 — Error handling
Dead Letter Queue — trong transaction
Messages không thể xử lý (deserialize fail, validation fail không thể retry) phải được ghi vào DLQ trong cùng transaction. Nếu ghi DLQ ngoài transaction, có thể DLQ ghi thành công nhưng offset không commit → message xuất hiện ở DLQ hai lần.
producer.beginTransaction();
try {
    for (ConsumerRecord<?, ?> record : records) {
        try {
            PaymentRequest req = deserialize(record.value());
            PaymentResult result = process(req);

            producer.send(new ProducerRecord<>(
                result.ok() ? "payment-processed" : "payment-failed",
                req.id(), toJson(result)
            ));

        } catch (DeserializeException | ValidationException e) {
            // Message không xử lý được → DLQ trong transaction
            Headers headers = buildDlqHeaders(record, e);
            producer.send(new ProducerRecord<>(
                "payment-dead-letter",
                null,           // partition = null
                record.key(),
                record.value(), // raw value để debug
                headers
            ));
            // Không throw — tiếp tục xử lý records khác trong batch
        }
    }

    producer.sendOffsetsToTransaction(offsets, consumer.groupMetadata());
    producer.commitTransaction();
    // Atomic: output + DLQ + offset → tất cả hoặc không có gì

} catch (KafkaException e) { ... }
DLQ headers quan trọng: Thêm dlq-reason, dlq-original-topic, dlq-original-partition, dlq-original-offset, dlq-timestamp để team ops có thể trace và replay sau.
Phase 6 — Production
Kubernetes deployment — unique ID per pod
Rolling deployment phải đảm bảo pod cũ shutdown hoàn toàn trước khi pod mới nhận traffic. Nếu không, hai pods cùng transactional.id chạy song song → fence liên tục.

  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # tối đa 1 pod mới trước khi terminate pod cũ
      maxUnavailable: 0  # không terminate cũ trước khi mới ready

  template:
    spec:
      containers:
      - name: payment-processor
        env:
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name  # inject pod name

        resources:
          limits:
            memory: "1Gi"   # OOM → crash → zombie risk nếu quá thấp
            cpu: "1000m"

        livenessProbe:
          httpGet: { path: /health/live, port: 8080 }
          initialDelaySeconds: 30
          failureThreshold: 3

        lifecycle:
          preStop:
            exec:
              command: ["/bin/sh", "-c", "sleep 15"]
              # Đủ thời gian hoàn thành transaction hiện tại

      # > transaction.timeout.ms để không bị kill giữa chừng

Pre-production checklist
  • Broker: transaction.state.log.replication.factor = 3
  • Broker: transaction.state.log.min.isr = 2
  • Broker: offsets.topic.replication.factor = 3
  • Topics: min.insync.replicas = 2
  • Producer: enable.idempotence = true
  • Producer: transactional.id = unique per instance
  • Producer: transaction.timeout.ms < broker max
  • Producer: initTransactions() gọi trước subscribe()
  • Consumer: isolation.level = read_committed
  • Consumer: enable.auto.commit = false
  • Consumer: max.poll.interval.ms > worst-case batch processing time
  • Code: sendOffsetsToTransaction() thay vì commitSync()
  • Code: ProducerFencedException → shutdown, không retry
  • Code: KafkaException → abortTransaction() → resetToLastCommitted()
  • Code: DLQ write nằm trong transaction
  • Test: producer crash mid-transaction → không có partial data
  • Test: broker crash → idempotent retry không duplicate
