01 Hiểu đúng trước khi làm

Trước khi tạo subagent, cần hiểu rõ bản chất của nó để không bị dùng sai chỗ.

🤖 Subagent là gì?
  • Một Claude instance riêng biệt
  • Có system prompt tùy chỉnh
  • Context window độc lập (200k)
  • Danh sách tools bị giới hạn
  • Chạy task xong → trả summary về
❌ Subagent KHÔNG phải là gì?
  • Không phải chatbot khác
  • Không nhớ giữa các lần gọi
  • Không phải "plugin" hay extension
  • Không thay thế CLAUDE.md
  • Không tốt hơn nếu dùng nhiều quá

Khi nào NÊN dùng Subagent?

Tình huốngLý do
Task tạo ra output rất lớn (logs, test results)Giữ main context sạch
Task tự chứa hoàn toàn, không cần back-and-forthIsolate context tốt hơn
Cần giới hạn tool (reviewer chỉ được Read, không Write)An toàn, kiểm soát rõ hơn
Có thể chạy song song nhiều taskTăng tốc đáng kể
Task lặp đi lặp lại, cần cùng chất lượng mỗi lầnReproducibility
⚠️
Mỗi subagent bắt đầu từ đầu — không có memory từ lần gọi trước. Nếu cần persistent memory, dùng Memory MCP hoặc ghi file ra disk để subagent đọc lại.

03 Giải phẫu file Subagent

Mỗi subagent là một file .md với 2 phần: YAML frontmatter (cấu hình) và Markdown body (system prompt).

.claude/agents/code-reviewer.md 
---
name: code-reviewer            ← ID duy nhất, dùng để gọi thủ công
description: |               ← Claude đọc cái này để TỰ ĐỘNG gọi agent
  Expert code reviewer for TypeScript/React.
  Use proactively after writing or modifying
  any source code. Reviews for security,
  performance, and maintainability.
tools: Read, Grep, Glob       ← Chỉ được Read, KHÔNG thể sửa file
model: sonnet                 ← haiku (rẻ/nhanh) | sonnet | opus
---

# Phần này là System Prompt của agent

Bạn là senior code reviewer với 10 năm kinh nghiệm.
Chuyên về TypeScript, React, Node.js, và security.

## Quy trình review (luôn theo thứ tự này):

1. Đọc toàn bộ file được chỉ định
2. Kiểm tra security vulnerabilities (SQL injection, XSS, auth bypass)
3. Đánh giá performance (unnecessary re-renders, N+1 queries)
4. Kiểm tra type safety và proper error handling
5. Đánh giá readability và naming conventions
6. Kiểm tra edge cases chưa xử lý

## Output format:

Trả về structured report với:
- **CRITICAL**: Vấn đề phải fix ngay (security, data loss)
- **WARNING**: Vấn đề nên fix (performance, best practices)
- **SUGGESTION**: Cải thiện optional

Với mỗi vấn đề: line number + explanation + code fix example.

## Quan trọng:

- Honest và critical — không praise vô nghĩa
- Specific với line numbers, không nói chung chung
- Nếu code tốt → nói rõ tại sao tốt

Các fields trong YAML frontmatter

FieldBắt buộc?Giá trịTác dụng
nameBắt buộckebab-case stringID để gọi thủ công: "Use the code-reviewer..."
descriptionBắt buộcText mô tảClaude đọc để quyết định có tự động gọi không
toolsNên cóComma-separated listGiới hạn tools được phép dùng
modelTùy chọnhaiku / sonnet / opusChọn model — mặc định là sonnet
💡
Description là quan trọng nhất. Claude đọc description để quyết định TỰ ĐỘNG gọi agent hay không. Viết rõ: khi nào dùng agent này, đầu vào là gì, đầu ra là gì. Dùng cụm từ action-oriented như "Use after...", "Invoke when...".

04 Ví dụ 1: Code Reviewer Agent

Scenario: Bạn vừa code xong feature mới và muốn review trước khi tạo PR.

Bước 1: Tạo agent

.claude/agents/code-reviewer.md 
---
name: code-reviewer
description: Expert code review specialist. Proactively
  reviews code for quality, security, and maintainability.
  Use immediately after writing or modifying code.
tools: Read, Grep, Glob
model: sonnet
---

Bạn là senior code reviewer. Khi được gọi:

1. Đọc kỹ code được chỉ định
2. Kiểm tra security vulnerabilities
3. Đánh giá performance và readability
4. Báo cáo với 3 mức: CRITICAL / WARNING / SUGGESTION

Với mỗi issue: dòng số + giải thích + ví dụ cách fix.
Honest và critical. Không vague.

Bước 2: Gọi agent

# Gọi thủ công — explicit
Use the code-reviewer subagent to review src/auth/login.ts

# Hoặc Claude tự gọi khi bạn nói:
Review the login handler I just wrote

Bước 3: Agent làm việc (bạn thấy trong terminal)

▶ Spawning subagent: code-reviewer
  Reading: src/auth/login.ts
  Reading: src/auth/types.ts
  Analyzing security patterns...
▶ Subagent complete — returning summary

Bước 4: Nhận kết quả trong main chat

# Agent trả về summary, KHÔNG dump toàn bộ output vào main chat:

Code Review: src/auth/login.ts

CRITICAL (1):
- Line 23: Password so sánh bằng == thay vì bcrypt.compare()
  Fix: const valid = await bcrypt.compare(password, hash)

WARNING (2):
- Line 45: SQL query dễ bị injection nếu input chưa sanitize
- Line 67: Missing rate limiting cho login attempts

SUGGESTION (1):
- Line 12: Nên extract magic string "7d" thành JWT_EXPIRY constant

05 Ví dụ 2: Debugger Agent

Scenario: Test đang fail và bạn không biết tại sao. Thay vì dump toàn bộ error vào main chat, delegate cho Debugger agent.

Bước 1: Tạo agent

.claude/agents/debugger.md 
---
name: debugger
description: Debugging specialist for errors, test failures,
  and unexpected behavior. Use proactively when encountering
  any errors, exceptions, or failing tests.
tools: Read, Edit, Bash, Grep, Glob
model: sonnet
---

Bạn là expert debugger. Root cause analysis là ưu tiên số 1.

Khi được gọi:
1. Đọc error message và stack trace
2. Tìm file liên quan qua Grep/Glob
3. Đặt hypotheses về nguyên nhân
4. Implement minimal fix (không refactor rộng)
5. Chạy test để verify fix

Với mỗi bug, báo cáo:
- Root cause (không phải symptom)
- Evidence (dòng code cụ thể)
- Fix đã apply
- Cách prevent tương lai

Bước 2: Gọi agent với context lỗi

# Cách 1: Paste error trực tiếp
Use the debugger subagent to fix this:
TypeError: Cannot read properties of undefined (reading 'userId')
  at getUserProfile (src/users/service.ts:45:22)

# Cách 2: Mô tả triệu chứng
The test "should return user profile" is failing. 
Use the debugger subagent to find and fix the issue.
💡
Debugger agent có quyền Edit + Bash — nó có thể tự chạy test và fix bug. Code reviewer thì không. Đây là ví dụ tốt về scoping tools đúng cách.

06 Ví dụ 3: Pipeline SDLC — Agents nối tiếp nhau

Scenario: Triển khai feature từ ticket đến PR hoàn chỉnh, với mỗi bước được xử lý bởi một agent chuyên biệt.

pm-spec
Đọc ticket
→ Viết spec
→
architect
Validate design
→ Tạo ADR
→
implementer
Code + Tests
→ Commits
→
code-reviewer
Final review
→ PR ready

Setup: 3 file agents cần tạo thêm

.claude/agents/pm-spec.md 
---
name: pm-spec
description: Product manager agent. Reads ticket/requirements
  and produces technical spec. Use at start of any new feature.
tools: Read, Grep, WebFetch   ← Chỉ đọc, không viết code
model: sonnet
---

Bạn là technical PM. Đọc requirements và output:
1. User stories với acceptance criteria rõ ràng
2. Technical constraints cần lưu ý
3. Files/modules cần modify (dựa trên codebase scan)
4. Estimated complexity: S/M/L

Lưu output vào docs/specs/{feature-name}.md
.claude/agents/implementer.md 
---
name: implementer
description: Implementation agent. Takes approved spec and
  implements the feature with tests. Use after spec is approved.
tools: Read, Write, Edit, Bash, Grep, Glob
model: sonnet
---

Bạn là senior developer. Nhận spec đã approved và:
1. Đọc spec từ docs/specs/
2. Implement feature theo conventions trong CLAUDE.md
3. Viết unit tests (coverage ≥ 80%)
4. Chạy pnpm test — fix nếu có lỗi
5. Tạo commit với message chuẩn
6. Báo cáo: files changed + test results

Gọi pipeline bằng một prompt

# Orchestrate toàn bộ pipeline từ main chat:
Implement the user avatar upload feature:

1. Use the pm-spec subagent to create a spec from this requirement:
   "Users should be able to upload and crop a profile avatar.
    Max 5MB, JPG/PNG only, store in S3."

2. After spec is ready, use the implementer subagent to build it.

3. Finally, use the code-reviewer subagent to review the implementation.
ℹ️
Pipeline này là sequential — mỗi agent đợi agent trước hoàn thành. Đây là đúng vì implementer cần spec từ pm-spec. Khi không phụ thuộc nhau, dùng parallel (xem ví dụ 4).

07 Ví dụ 4: Parallel Review — Tất cả chạy cùng lúc

Scenario: Review một PR với nhiều khía cạnh khác nhau — security, performance, tests — tất cả chạy song song, giảm thời gian từ 5 phút xuống 30 giây.

Setup: Tạo 2 agent bổ sung

.claude/agents/security-auditor.md 
---
name: security-auditor
description: Security specialist. Scans for vulnerabilities,
  auth issues, injection attacks, secrets exposure. Use when
  reviewing authentication, API endpoints, or database queries.
tools: Read, Grep, Glob
model: sonnet
---

Security auditor chuyên về OWASP Top 10.
Kiểm tra: SQL injection, XSS, CSRF, auth bypass,
secrets in code, insecure dependencies, input validation.
Báo cáo theo mức độ: CRITICAL → HIGH → MEDIUM → LOW.
.claude/agents/test-writer.md 
---
name: test-writer
description: Test generation specialist. Writes missing unit
  and integration tests. Use when code coverage is insufficient
  or after implementing new features.
tools: Read, Write, Bash, Glob
model: haiku                  ← Haiku đủ tốt, rẻ hơn 5x
---

Test writer cho Vitest + TypeScript.
Đọc source file → phân tích logic → viết tests cover:
- Happy path
- Error cases
- Edge cases (null, empty, overflow)
Target: ≥ 85% coverage.
Đặt test file bên cạnh source: foo.ts → foo.test.ts

Gọi tất cả song song bằng 1 prompt

# Keyword "simultaneously" / "in parallel" / "at the same time"
# → Claude hiểu là chạy song song

Run these 3 subagents simultaneously on the payment module:

1. security-auditor on src/payments/
2. code-reviewer on src/payments/checkout.ts
3. test-writer to generate missing tests for src/payments/utils.ts

Kết quả — 3 reports trả về main chat

▶ Spawning 3 subagents in parallel...
  [security-auditor] Reading src/payments/ (8 files)
  [code-reviewer] Reading src/payments/checkout.ts
  [test-writer] Reading src/payments/utils.ts
  ...
  [test-writer] Complete ✓ — 12 tests written, 87% coverage
  [code-reviewer] Complete ✓ — 2 warnings, 1 suggestion
  [security-auditor] Complete ✓ — 1 HIGH, 2 MEDIUM findings

━━━ Summary from 3 agents ━━━
Security: 1 HIGH issue — missing CSRF token on /checkout endpoint
Code Quality: Deprecated API usage on line 45
Tests: Generated checkout.utils.test.ts — 12 tests, 87% coverage

09 Tool permissions — Scoping đúng cách

Đây là một trong những điều quan trọng nhất. Mỗi agent chỉ nên có đúng tools nó cần.

Loại agentTools nên cóTại sao
Reviewer, Auditor Read Grep Glob Chỉ đọc — không bao giờ sửa file
Researcher, Analyst Read Grep Glob WebFetch Tìm kiếm docs, không sửa code
Implementer, Developer Read Write Edit Bash Glob Grep Cần toàn quyền để code và test
Doc writer Read Write WebFetch Đọc code + viết docs, không chạy code
Debugger Read Edit Bash Grep Cần chạy test để verify fix
⚠️
Nếu bỏ trống field tools, agent kế thừa tất cả tools của main session (kể cả MCP). Đây là nguy hiểm nếu bạn có MCP servers nhạy cảm. Luôn whitelist tools cụ thể.

10 Lỗi thường gặp & cách fix

❌ Agent không tự động được gọi

Nguyên nhân: description quá chung chung hoặc quá ngắn.

# ❌ Bad description:
description: Helps with code

# ✅ Good description:
description: |
  Reviews TypeScript/React code for security vulnerabilities,
  PEP 8 compliance, and performance issues. Use proactively
  after writing or modifying any source files.

❌ Agent làm quá nhiều hoặc quá ít

Nguyên nhân: system prompt không đủ chi tiết. Fix: thêm step-by-step workflow và output format cụ thể.

❌ Context window vẫn bị ngốn nhiều

Nguyên nhân: agent return quá nhiều detail. Fix: yêu cầu agent chỉ trả summary trong system prompt.

# Thêm vào system prompt của agent:
Return ONLY a concise summary (max 200 words).
Do NOT include raw output, logs, or verbose details.
Put full details in a file if needed.

❌ Nhiều agents hơn thì tốt hơn?

🚫
Sai. Sau 3-4 agents, năng suất của bạn thường giảm vì overhead quản lý. Bắt đầu với 1 agent, làm quen, rồi mới thêm.

❌ Chaining agents làm tốn token nhanh

Mỗi agent có context riêng, nhưng kết quả trả về vẫn tốn context của main session. Với task cần 20+ agents liên tiếp, xem xét dùng Agent Teams (feature nâng cao hơn Subagents).
