Upgrade từ Subagents lên Agent Teams — khi bạn cần nhiều AI làm việc với nhau, không chỉ làm việc song song.

⚠️ Tính năng experimental, bật thủ công — xem hướng dẫn setup bên dưới

01 Agent Teams vs Subagents — Khác nhau ở đâu?

Đây là câu hỏi quan trọng nhất. Hiểu sai điều này sẽ khiến bạn dùng sai tool.

Subagents Agent Teams
Context Context riêng, trả kết quả về main Context riêng, hoàn toàn độc lập
Giao tiếp Chỉ report về main agent Teammates nhắn tin trực tiếp cho nhau
Coordination Main agent quản lý tất cả Shared task list, tự phân việc
Token cost Thấp hơn Cao hơn đáng kể
Dùng khi Task đơn, chỉ cần kết quả cuối Task phức tạp, cần thảo luận và phối hợp
Analogy Thuê thợ chuyên, nhận hàng xong trả tiền Làm việc cùng một team cố định
💡
Insight cốt lõi: Subagents không thể nói chuyện với nhau — chúng chỉ báo cáo về main agent. Agent Teams thì khác: teammate A có thể nhắn cho teammate B, challenge lý thuyết của nhau, chia sẻ findings — giống team người thật.

02 Bật Agent Teams

Agent Teams là experimental và tắt theo mặc định. Cần enable thủ công.

1
Mở file settings.json của Claude Code
# macOS / Linux
~/.claude/settings.json

# Nếu chưa có file, tạo mới:
mkdir -p ~/.claude && touch ~/.claude/settings.json
2
Thêm env variable
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
⚠️
Đây là experimental — có known limitations về session resumption, task coordination, và shutdown behavior. Không dùng cho production-critical workflows chưa test kỹ.
3
(Tùy chọn) Cài tmux hoặc iTerm2 cho split panes
Không bắt buộc — in-process mode mặc định hoạt động ở mọi terminal. Nhưng tmux/iTerm2 cho phép thấy output tất cả teammates đồng thời.
# macOS — chọn một trong hai:
brew install tmux
# hoặc download iTerm2 từ iterm2.com

# Sau đó set trong settings.json (nếu muốn):
{
  "teammateMode": "tmux"
}
4
Kiểm tra đã bật thành công
# Khởi động Claude Code và thử:
claude

# Trong session, hỏi Claude:
Can you create a small agent team to test if the feature is enabled?
Nếu Claude bắt đầu nói về "spawning teammates" thay vì subagents, bạn đã bật thành công.

05 Ví dụ 1: Debug với competing hypotheses

🔍
Bug khó — nhiều giả thuyết cùng lúc
5 teammates, mỗi người investigate một theory khác nhau, rồi challenge nhau

Đây là use case mạnh nhất của Agent Teams. Debug tuần tự bị anchoring — lần đầu tìm ra một theory, các bước sau bị bias theo nó. Nhiều investigators độc lập chạy adversarial debate sẽ tìm ra root cause nhanh hơn nhiều.

Bước 1: Mô tả bug cho Lead

Users report the checkout process fails intermittently.
Error only happens ~15% of the time, no consistent pattern in logs.

Create an agent team to investigate competing hypotheses:
1. Race condition in payment gateway callback
2. Database connection pool exhaustion under load
3. Session timeout during 3D Secure redirect
4. Third-party shipping API latency spike
5. Memory leak in cart serialization

Have them investigate in parallel, then debate each other's findings.
Update a shared findings doc with whatever consensus emerges.

Bước 2: Lead tạo team và spawn 5 teammates

▶ Creating team: checkout-debug
▶ Creating task list...
  Task 1: Investigate race condition hypothesis
  Task 2: Investigate DB connection pool hypothesis
  Task 3: Investigate session timeout hypothesis
  Task 4: Investigate shipping API hypothesis
  Task 5: Synthesize findings (blocked by 1-4)
▶ Spawning 5 teammates...

Bước 3: Teammates hoạt động (bạn thấy trong terminal)

[teammate-1] Scanning src/payments/ for async operations...
[teammate-2] Checking db/pool.ts and connection config...
[teammate-3] Tracing session lifecycle in auth middleware...
[teammate-4] Fetching shipping API logs from last 7 days...

# Teammates gửi findings cho nhau qua mailbox:
[teammate-1 → teammate-2]: Found uncaught Promise in payment callback.
  Could correlate with your connection drops? Check timing.
[teammate-2 → teammate-1]: Connection drops happen AFTER payment callback,
  not during. Your race condition theory is more likely.

Bước 4: Tương tác trực tiếp với teammate

# Bạn có thể nhắn thẳng cho teammate-1 (không qua lead):
[Select teammate-1 bằng Shift+Up/Down]
Can you also check if the issue appears in staging environment logs?

Bước 5: Nhận kết quả tổng hợp

━━━ Consensus from checkout-debug team ━━━

Root cause: Race condition in payment callback (Confidence: HIGH)

Evidence:
- teammate-1: Uncaught Promise at payments/gateway.ts:127
- teammate-2: DB drops correlate temporally with callback (not causal)
- teammate-3: Session valid throughout — ruled out
- teammate-4: Shipping API latency normal — ruled out

Fix: Add proper await + error boundary at gateway.ts:127
Test: 500 concurrent checkout simulations to verify

Bước 6: Shutdown và cleanup

# Nói với Lead:
The race condition is confirmed fixed. Please shut down all teammates
and clean up the checkout-debug team.

06 Ví dụ 2: Code Review đa chiều

🔬
Review PR từ 3 góc độ độc lập
Security + Performance + Architecture — mỗi người nhìn từ lens khác nhau

Single reviewer thường bị gravitating về một loại issue. Chia review criteria cho 3 teammates độc lập, mỗi người tập trung hoàn toàn vào domain của mình.

Spawn prompt

Create a code review team for PR #247 (auth module refactor).

Spawn 3 reviewer teammates:

1. security-reviewer: Review src/auth/ for OWASP Top 10 vulnerabilities.
   Focus: token handling, session management, input validation, secrets exposure.
   Rate issues: CRITICAL / HIGH / MEDIUM / LOW.

2. performance-reviewer: Review for performance issues.
   Focus: N+1 queries, unnecessary re-renders, memory leaks, slow algorithms.
   Include estimated impact where possible.

3. architecture-reviewer: Review design decisions.
   Focus: SOLID principles, coupling, cohesion, testability, future extensibility.

Have them share findings with each other. If they disagree on priority,
debate and reach consensus before reporting to me.
💡
Điểm khác biệt so với Subagents: reviewers debate với nhau. Nếu security-reviewer tìm ra một issue nhưng architecture-reviewer cho rằng fix sẽ break design pattern, họ sẽ thảo luận trực tiếp trước khi báo về cho bạn.

Bật Plan Approval cho review nghiêm túc

# Nếu muốn approve plan của từng reviewer trước khi họ bắt đầu:
Spawn 3 reviewer teammates. Require plan approval before each starts.

# Lead sẽ hỏi bạn xác nhận plan của từng teammate:
# "security-reviewer plans to: read all files in src/auth/,
#  run grep for hardcoded secrets, check JWT implementation..."
# Approve? (y/n)

07 Ví dụ 3: Full-stack Feature — Phân chia theo layer

🏗️
Implement feature spanning frontend + backend + tests
3 teammates mỗi người own một layer, không conflict nhau

Nguyên tắc phân chia task cho Agent Teams

Quan trọng nhất: file ownership phải rõ ràng. Hai teammates edit cùng file sẽ gây overwrites và conflicts.

TeammateFiles sở hữuKhông được đụng vào
frontend-devsrc/components/**, src/pages/**src/api/**, src/db/**
backend-devsrc/api/**, src/services/**src/components/**, src/pages/**
test-engineertests/**src/** (chỉ đọc)

Spawn prompt với file ownership rõ ràng

Implement the "user notification preferences" feature.
Spec is at docs/specs/notification-prefs.md.

Create a team with 3 teammates, each owning a distinct layer:

1. frontend-dev:
   - Owns: src/components/NotificationPrefs/, src/pages/settings/
   - Build the UI with form controls and validation
   - Wait for backend-dev's API contract before implementing fetch calls

2. backend-dev:
   - Owns: src/api/notifications/, src/services/notifications/
   - Implement REST endpoints per the spec
   - Write API contract doc at docs/api/notification-prefs.md

3. test-engineer:
   - Owns: tests/unit/notifications/, tests/e2e/settings/
   - Read-only access to src/
   - Write tests after frontend and backend signal completion

Dependencies: test-engineer is blocked by frontend-dev AND backend-dev.
frontend-dev is blocked by backend-dev's API contract doc.

Task dependency tracking tự động

# Claude tạo task list với auto-unblocking:
Task 1: Write API contract  [owner: backend-dev]   → IN PROGRESS
Task 2: Implement UI        [owner: frontend-dev]  → BLOCKED by Task 1
Task 3: Implement API       [owner: backend-dev]   → IN PROGRESS
Task 4: Write tests         [owner: test-engineer] → BLOCKED by Task 2, 3

# Khi Task 1 hoàn thành → Task 2 tự động UNBLOCKED
# frontend-dev self-claim Task 2 ngay lập tức
⚠️
Quy tắc file ownership: Nếu 2 teammates edit cùng file → conflict và overwrites. Luôn phân chia tasks sao cho mỗi file chỉ có một teammate owner tại một thời điểm.

Plan Approval — Kiểm soát trước khi thực thi

Dùng khi task có rủi ro cao — teammate phải submit plan, bạn approve/reject trước khi họ bắt tay vào làm.

# Require approval trong spawn prompt:
Spawn a refactoring teammate for the auth module.
Require plan approval before they make any changes.

# Chỉ định criteria approve:
Only approve plans that:
- Include test coverage for every changed function
- Do NOT modify the database schema
- Estimate less than 5 files changed

Delegate Mode — Lead chỉ coordinate, không code

Nhấn Shift+Tab để restrict lead vào coordination-only mode. Ngăn chặn vấn đề phổ biến: lead bị distracted và tự implement thay vì đợi teammates.

# Hoặc nhắc Lead trong prompt:
Create the team and delegate all implementation to teammates.
Your job is only to spawn, coordinate, and synthesize.
Do NOT write any code yourself. Wait for teammates.

Shutdown đúng cách

# Bước 1: Yêu cầu Lead shutdown từng teammate
Please shut down all teammates gracefully.

# Teammates có thể reject nếu đang ở giữa task quan trọng:
[teammate-2]: Cannot shut down — currently writing critical test.
  Will complete in ~2 minutes. Please wait.

# Bước 2: Sau khi tất cả shutdown xong, cleanup team
Clean up the team.

# Lead tự kiểm tra active teammates trước khi cleanup
# Nếu còn teammate active → cleanup sẽ fail với error

09 Pitfalls thực tế & cách xử lý

Vấn đềBiểu hiệnFix
Lead tự implement thay vì delegate Lead bắt đầu viết code thay vì chờ teammates Bật Delegate mode (Shift+Tab) hoặc nhắc rõ trong prompt
Task status lag Task xong nhưng vẫn hiện "in progress", blocking dependent tasks Hỏi Lead kiểm tra task thực sự done chưa, update thủ công nếu cần
Session không resume được Sau /resume, Lead cố nhắn tin cho teammate không còn tồn tại Spawn fresh teammates sau khi resume
File conflicts Hai teammates edit cùng file → overwrites Phân chia file ownership rõ ràng ngay từ spawn prompt
Token cost vượt kỳ vọng Hóa đơn cao hơn nhiều so với dùng single agent Dùng Agent Teams chỉ khi task thực sự cần parallel specialists
⚠️
Không có nested teams. Teammates không thể spawn thêm teammates. Một lead quản lý một team tại một thời điểm. Thiết kế này là có chủ ý — ngăn infinite recursion và mất kiểm soát token cost.

10 Khi nào dùng gì?

Đây là câu hỏi thực tế quan trọng nhất. Đừng dùng Agent Teams chỉ vì nó "nghe có vẻ powerful hơn".

Tình huốngDùng gìLý do
Review code sau khi viết xong Subagent Task đơn, chỉ cần kết quả cuối
Fix một bug cụ thể Single session Sequential, không cần parallel
Debug bug intermittent khó tái hiện Agent Teams Multiple hypotheses song song, cần debate
Review PR từ nhiều góc độ (security + perf + arch) Agent Teams Reviewers cần share và challenge findings của nhau
Chạy tests, lint, format Subagent / Hook Routine task, single-tool, không cần coordination
Feature spanning frontend + backend + tests Agent Teams Clear file ownership, genuine parallel value
Research 3 approaches cho một architectural decision Agent Teams Teammates khám phá độc lập rồi debate
Viết documentation Subagent Sequential, không cần inter-agent communication
🎯
Rule of thumb từ Addy Osmani: "Let the problem guide the tooling, not the other way around. If a single agent in a focused session gets you there faster, use that. Agent Teams add coordination overhead and use significantly more tokens — they pay off only on larger, parallelizable work where teammates can operate independently."

Pyramid of tools — dùng từ dưới lên

                    ┌─────────────────┐
                    │   Agent Teams   │  ← Parallel, cần debate
                    │  (experimental) │    Cross-layer coordination
                    └────────┬────────┘
               ┌─────────────┴─────────────┐
               │         Subagents          │  ← Isolate context
               │   (stable, recommended)    │    Parallel tasks, no comms
               └─────────────┬─────────────┘
          ┌───────────────────┴───────────────────┐
          │          Single Session                │  ← 90% of daily work
          │   (cheapest, fastest, most reliable)   │
          └───────────────────────────────────────┘

Bắt đầu từ single session. Upgrade lên Subagents khi cần isolate context. Upgrade lên Agent Teams chỉ khi teammates thực sự cần communicate với nhau.
